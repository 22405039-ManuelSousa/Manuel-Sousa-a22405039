Modelação em Papel

O diagrama entidade-relação foi desenhado em papel antes de começar a implementação. Os modelos principais são: Projeto, Tecnologia, Competencia, Formacao, UnidadeCurricular, Licenciatura, Docente, TFC, MakingOf, Conquista, Artigo, Comentario e Avaliacao.

Arquitetura MVT — Componentes Django

Model: define a estrutura dos dados e as relações entre eles (ForeignKey, ManyToManyField).

View: recebe o pedido HTTP, consulta o modelo e decide o que mostrar.

Template: ficheiro HTML que recebe os dados da view e os apresenta ao utilizador.

Exemplo: quando alguém abre /projetos/, o Django encaminha para projetos_view, que consulta o modelo Projeto e devolve projetos.html com a lista preenchida.

Estrutura do Site

O site está organizado em 3 apps Django:

  • portfolio — tecnologias, projetos, TFCs, competências, formações, conquistas, making-of, sobre
  • escola — docentes, licenciaturas, unidades curriculares
  • artigos — artigos, comentários, avaliações, registo de autores
  • accounts — login, logout, magic link

Tecnologias Usadas

  • Django 6 — framework web Python (MVT)
  • PostgreSQL / Neon — base de dados em cloud
  • Cloudinary — armazenamento de ficheiros media
  • Whitenoise — servir ficheiros estáticos em produção
  • Gunicorn — servidor WSGI para produção
  • Railway — plataforma de deployment
  • GitHub — controlo de versões e CI/CD
  • HTML/CSS — templates e estilo

Arquitetura de Produção

A aplicação segue uma arquitetura profissional com separação de responsabilidades:

  • Base de dados: Neon (PostgreSQL serverless na cloud) — elimina a dependência de SQLite local
  • Media: Cloudinary — imagens e ficheiros guardados externamente, não no servidor
  • Estáticos: Whitenoise — serve CSS/JS diretamente pelo Django em produção
  • Servidor: Railway — deployment automático a cada push para o GitHub

Pipeline CI/CD

O fluxo de desenvolvimento e deployment funciona assim:

  1. Desenvolvimento local com runserver e base de dados Neon
  2. git add + git commit + git push para o GitHub
  3. Railway deteta o push automaticamente
  4. Railway faz build com Railpack (deteta Python/Django automaticamente)
  5. Executa gunicorn portfolio_project.wsgi:application
  6. Site atualizado em produção em a volta de 2 minutos
